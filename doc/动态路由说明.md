# 动态路由创建流程说明（vue-admin）

> 本文档聚焦 `apps/vue-admin` 内的动态路由创建与注入过程。默认前提：动态路由能力已收敛到 `vue-admin`，`ursacomponents` 仅作为 UI 组件库使用。

---

## 1. 目标与范围

本文回答 4 个问题：

1. 动态路由在哪些时机触发？
2. 动态路由如何由菜单数据转换为可注册路由？
3. 如何避免重复注册、刷新失效和登录回环？
4. 出现 404 / 白屏 / 刷新丢路由时如何排查？

范围仅覆盖 `apps/vue-admin/src`：

- `router/dynamic-routes.js`
- `router/index.js`
- `main.js`
- `composables/useLogin.js`
- `stores/user.js`

---

## 2. 入口文件与职责

### 2.1 `src/router/dynamic-routes.js`

动态路由能力入口，职责：

- 扫描 `views` 页面模块（`import.meta.glob`）
- 按菜单数据构建路由记录
- 提供 `initDynamicRoutes(router, menus)` 完成批量注入
- 提供图标解析（用于菜单展示）

> 约定：登录页不参与动态路由扫描。

### 2.2 `src/router/index.js`

路由总入口，职责：

- 注册静态路由（`/login`、`/` + `layout`）
- 在前置守卫中处理 token 校验
- 在“需要”时触发动态路由初始化

### 2.3 `src/main.js`

应用启动入口，职责：

- 初始化 Vue、Pinia、路由、全局组件
- 在挂载前读取持久化用户菜单并预加载动态路由

### 2.4 `src/composables/useLogin.js`

登录流程入口，职责：

- 登录成功后写入 token 与用户信息
- 立即执行 `initDynamicRoutes` 注入业务页面
- 记录 `hasLoadedAsyncRoutes=true`
- 跳转首个可访问菜单

### 2.5 `src/stores/user.js`

用户状态中心，职责：

- 保存 `userInfo`（含 `menus`）
- 维护 `hasLoadedAsyncRoutes`，防止重复注入
- 提供重置逻辑（登出/身份失效）

---

## 3. 动态路由触发时机（3 条主链路）

## 3.1 链路 A：应用冷启动预加载

触发位置：`main.js`

流程：

1. 创建应用并初始化 pinia/router。
2. 读取 `userStore.userInfo?.menus`。
3. 若菜单非空：
   - 调用 `initDynamicRoutes(router, menus)`
   - 调用 `setHasLoadedAsyncRoutes(true)`
4. `app.use(router)` 并 `mount`。

价值：刷新后可直接命中业务页，减少首次导航失败。

---

### 3.2 链路 B：登录成功后即时加载

触发位置：`useLogin.handleLogin`

流程：

1. 登录成功，拿到 `token + menus`。
2. 写 token、写 userStore。
3. 调用 `initDynamicRoutes(router, res.menus)`。
4. 标记已加载异步路由。
5. 计算首菜单路径并 `router.push(firstPath)`。

价值：首次登录后无需刷新即可进入业务页面。

---

### 3.3 链路 C：路由守卫兜底加载

触发位置：`router/index.js` 的前置守卫

典型触发条件：

- 已有 token，但动态路由尚未注入
- 页面直链访问导致 `to.matched` 不完整
- 刷新后内存路由丢失

流程：

1. 先校验登录态（无 token 跳 `/login`）。
2. 判断是否需要加载动态路由。
3. 读取菜单并调用 `initDynamicRoutes`。
4. 标记加载完成。
5. 使用 `next({ ...to, replace: true })` 让新路由立即生效。

价值：兜底覆盖刷新、直链、状态边界等场景。

---

## 4. `initDynamicRoutes` 建议实现步骤（标准化）

为便于维护，建议在 `dynamic-routes.js` 固化以下步骤：

1. **输入校验**
   - `router` 是否可用（具备 `addRoute/hasRoute`）
   - `menus` 是否数组

2. **菜单标准化**
   - 统一 `path/component/name/meta` 字段
   - 清洗空节点、重复节点

3. **菜单 -> 路由映射**
   - 仅映射可解析页面组件的菜单项
   - 对缺失 `name` 的路由生成稳定 fallback name

4. **去重注入**
   - `router.hasRoute(name)` 为 false 才执行 `addRoute`
   - 默认挂载到 `layout` 子路由

5. **结果返回**
   - 返回本次构建/注入的路由数组，便于日志与测试

---

## 5. 菜单字段约定（建议）

建议菜单节点至少包含：

```ts
{
  path: string;          // 路由路径（如 /dashboard 或 dashboard）
  component: string;     // 组件相对路径（如 dashboard/index）
  name?: string;         // 路由名称，可选
  meta?: {
    title?: string;
    icon?: string;
    hidden?: boolean;
  };
  children?: MenuNode[];
}
```

补充约束：

- `path + name` 应全局唯一
- `component` 必须命中 `import.meta.glob` 扫描结果
- `hidden` 只影响菜单展示，不应默认阻断路由注册

## 6. 时序图（文字版）

```text
[main.js] / [useLogin.js] / [router guard]
              │
              ├─ 读取 menus
              │
              ├─ initDynamicRoutes(router, menus)
              │    ├─ 校验输入
              │    ├─ 菜单标准化
              │    ├─ 构建 routes[]
              │    └─ addRoute(layout, route) 去重注入
              │
              └─ setHasLoadedAsyncRoutes(true)
                       │
                       └─ 导航继续（必要时 replace 当前路由）
```

## 7. 故障排查清单

### 7.1 404 / 空白页

1. `component` 路径是否与 `views` 实际路径一致
2. `import.meta.glob` 是否包含目标文件
3. `path` 是否冲突或被错误拼接
4. 动态路由是否实际注入（查看路由表/日志）

### 7.2 登录后被打回登录页

1. token 是否成功持久化
2. `userInfo.menus` 是否为空
3. guard 是否在菜单未就绪时提前重定向
4. `hasLoadedAsyncRoutes` 状态是否被意外重置

### 7.3 刷新后业务页失效

1. Pinia 持久化插件是否生效
2. `main.js` 的预加载逻辑是否执行
3. guard 兜底链路是否生效

---

## 8. 维护建议

1. 把动态路由逻辑继续收敛在 `src/router/dynamic-routes.js`，避免散落。
2. 为“菜单->路由映射”增加单元测试（含异常菜单样例）。
3. 将调试日志改为环境变量开关，默认关闭。
4. 在登录、刷新、直链三种场景做回归检查脚本。
