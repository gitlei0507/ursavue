# 项目结构与功能说明

## 1. 项目整体架构

本仓库是一个基于 **pnpm workspace** 的 Monorepo，包含：

- `apps/vue-admin`：后台管理应用
- `packages/ursacomponents`：通用 UI 组件库

---

## 2. 目录结构

```text
ursavue/
├─ apps/
│  └─ vue-admin/
│     ├─ src/
│     │  ├─ api/                # 接口层（用户、SNMP 等）
│     │  ├─ composables/        # 业务组合式逻辑（登录、用户管理等）
│     │  ├─ config/             # 布局配置
│     │  ├─ layout/             # 主框架布局
│     │  ├─ router/             # 路由（静态路由 + 动态路由装配）
│     │  ├─ stores/             # Pinia 状态管理（用户、菜单、动态路由状态）
│     │  ├─ utils/              # 基础工具（鉴权、请求封装、菜单工具）
│     │  ├─ views/              # 页面视图
│     │  ├─ App.vue
│     │  └─ main.js             # 应用启动入口
│     ├─ doc/                   # 业务文档
│     └─ vite.config.js
└─ packages/
   └─ ursacomponents/
      ├─ src/
      │  ├─ components/         # 通用 UI 组件
      │  ├─ router/             # 路由相关通用 toolkit/guard/icon 等能力
      │  ├─ utils/              # 组件库工具
      │  ├─ styles/
      │  └─ index.js            # 组件库统一导出入口
      └─ doc/                   # 组件库文档（VitePress）
```

## 3. 模块职责说明

### 3.1 `apps/vue-admin`（业务应用层）

#### 1）启动与初始化
- `src/main.js`
  - 初始化 Vue 应用、Pinia、Element Plus、路由。
  - 应用挂载前读取用户菜单并执行动态路由预加载。

#### 2）路由与鉴权
- `src/router/index.js`
  - 维护静态路由（如登录页、layout 壳路由）。
  - 使用 `setupUrsaRouterGuard` 绑定登录态校验与动态路由初始化时机。
- `src/router/dynamic-routes.js`
  - **动态路由创建的业务入口（迁移后的核心变化）**。
  - 扫描 `views` 页面模块，基于菜单数据构建并注册路由。

#### 3）状态管理
- `src/stores/user.js`
  - 管理用户信息与菜单树。
  - 管理 `hasLoadedAsyncRoutes`，避免重复注入动态路由。

#### 4）业务逻辑层
- `src/composables/useLogin.js`
  - 登录成功后：写入 token、写入用户信息、初始化动态路由、跳转首个可访问菜单。
- `src/composables/useUser.js`
  - 用户模块增删改查交互逻辑封装。

#### 5）基础设施
- `src/utils/request/index.js`
  - axios 实例、请求/响应拦截、统一异常提示。
- `src/utils/menu/menu.js`
  - 菜单树构建与首路径查找；由业务侧调用，内部复用组件库通用实现。

#### 6）界面骨架
- `src/layout/index.vue`
  - 组合 `UrsaMenu`、`UrsaNavBar`、`UrsaTagsView` 与主内容区。

---

### 3.2 `packages/ursacomponents`（通用能力层）

#### 1）通用 UI 组件
- 菜单、导航栏、标签页、搜索、表格等可复用组件。

#### 2）组合式能力
- `useUrsaMenu`、`useUrsaSearch`、`useUrsaTable` 等逻辑复用函数。

#### 3）路由工具能力（工具层，而非业务编排层）
- 导出菜单树、图标解析、路由 guard/toolkit 等通用能力。
- 不持有具体业务菜单来源、页面组织、权限策略细节。

---

## 4. 动态路由迁移后的设计价值

将动态路由创建迁移到 `vue-admin` 后，形成了更清晰的分层：

1. **职责解耦**：
   - `vue-admin` 负责“业务编排”（菜单来源、页面扫描、路由注入时机）。
   - `ursacomponents` 负责“通用能力输出”（组件、工具、基础 guard）。

2. **可维护性提升**：
   - 业务路由变更只改应用层，不强依赖组件库发版。

3. **复用性提升**：
   - 组件库可被不同后台项目复用，各项目保留自身路由策略。

4. **风险隔离**：
   - 高频业务变更不污染基础库，降低跨项目连带影响。

## 5. 关键运行流程

### 5.1 应用冷启动流程
1. `main.js` 初始化应用与插件。
2. 从 `userStore` 读取持久化菜单。
3. 若存在菜单，调用 `initDynamicRoutes(router, menus)` 预加载动态路由。
4. 挂载路由并启动应用。

### 5.2 登录后动态路由流程
1. `useLogin.handleLogin` 调用登录接口。
2. 存储 token 与用户信息。
3. 调用 `initDynamicRoutes(router, res.menus)` 完成动态路由注入。
4. 标记 `hasLoadedAsyncRoutes=true`。
5. 跳转到首个可访问菜单路径。

### 5.3 路由守卫流程
1. `router/index.js` 调用 `setupUrsaRouterGuard`。
2. Guard 中判断 token、用户信息与动态路由加载状态。
3. 必要时触发动态路由初始化，保证刷新/直链场景可访问。

## 6. 维护建议

1. **保持分层边界**
   - 业务路由规则、菜单字段适配、页面路径规范放在 `vue-admin`。
   - 组件库仅沉淀稳定且无业务语义的能力。

2. **约束菜单 schema**
   - 建议在 `vue-admin` 为菜单数据增加 schema 校验（字段完整性、类型、路径规范）。

3. **增强可观测性**
   - 为动态路由注入增加可控日志开关与异常上报，便于线上排查。

4. **补齐测试**
   - 为菜单→路由映射、首菜单路径选择、guard 行为添加单测或集成测试。
